## 一、项目概述

本次实训我独立完成了一个**全栈个人博客系统**，采用 **Vue 3 + Express + SQLite** 技术栈，涵盖前端、后端、数据库三大模块。项目包含 8 个前端路由页面、30+ 后端 RESTful API 接口、8 张数据库表，实现了文章 CRUD、评论嵌套回复、点赞收藏、分类标签搜索、暗色模式、公告系统、友链管理、关于页面等完整功能。整个项目经历 13 次 Git 提交，代码托管于 GitHub，最终成功部署至腾讯云 CloudBase（前端静态托管 + 后端 CloudRun 容器），实现了全功能线上访问。

## 四、遇到的最大困难及解决过程

### 困难一：部署平台的反复尝试

最大的困难出现在部署环节。最初尝试使用 EdgeOne Pages 部署，反复遇到超时问题，共尝试 5 次均失败。排查过程：

1. **初步怀疑文件过大**：dist 目录 12MB（10 张封面图占 10.9MB）
2. **写 Python 脚本压缩图片**：用 Pillow 将图片从 800px 宽 + 优化保存，体积降到 4.1MB（-62%），但仍有超时
3. **排除法定位根因**：去掉图片后纯代码 1.21MB 仍然报错，查错误日志发现真正原因是 `Zone is not active`——EdgeOne Pages Zone 未激活
4. **最终方案**：切换至腾讯云 CloudBase，前端使用静态网站托管，后端使用 CloudRun 容器模式部署

### 困难二：后端容器化部署

将原本地运行的 Express + SQLite 服务迁移到 CloudRun 容器需要解决：
- SQLite 在容器中的持久化问题
- 数据库种子的自动初始化
- 前端 API 地址的动态切换

**解决方案**：
- 编写 Dockerfile（Node 18 + 自动启动）
- 创建 seed-db.js 脚本，启动时自动检测并填充 6 篇文章、3 条评论、用户、友链等测试数据
- 封装 api.js 实现开发环境用 Vite 代理、生产环境用云后端地址

### 经验总结

遇到问题不能盲目重试，要**逐步缩小范围定位根因**。部署调试需要耐心和系统性排查思路。

## 五、对 AI 辅助编程的心得体会

1. **AI 是超级加速器但不是替代品**：CSS 变量系统、深色模式、批量重构这些工作 AI 完成得非常快（几小时工作量几分钟搞定），但生成的代码需要仔细审查。例如 AI 在 articles.js 中写的路由关键字过滤逻辑 `['categories', 'hot'].includes(id)` 存在潜在冲突。

2. **Prompt 质量决定输出质量**：将课程考核要求的 docx 文件直接给 AI，AI 就能按照要求生成符合规范的代码和文档。这说明把约束条件明确告知 AI 非常重要。

3. **AI 只能辅助，不能完全依赖**：数据库迁移、密码加密逻辑、评论嵌套树构建、Docker 容器化等核心业务逻辑，需要自己理解原理后再让 AI 实现，否则出问题时无法独立排查。

4. **多轮迭代优于一次性生成**：本项目通过 8 轮对话逐步完善，每轮聚焦一个模块，既保持代码质量又避免上下文溢出。

5. **AI 在部署环节的优势明显**：CloudBase 相关技能的自动加载，让 Dockerfile 编写、CloudRun 部署配置、seed-db.js 脚本等操作变得高效。

## 七、项目亮点

1. **CSS 变量设计系统**：20+ 变量定义在 `:root`，一键切换深色模式，全局风格统一
2. **评论嵌套树结构**：内存构建嵌套树，时间复杂度 O(n)，支持无限层级回复
3. **数据库自动迁移**：增量迁移 + 错误容忍，兼容旧数据
4. **自动密码升级**：旧明文密码验证通过后自动升级为 bcrypt
5. **路由守卫 + 登录重定向**：未登录自动跳转登录页，登录后回到目标页
6. **Toast 插件化**：provider/inject 模式全局可用，统一操作反馈
7. **容器化部署**：Dockerfile + 自动种子，实现一键云端部署
